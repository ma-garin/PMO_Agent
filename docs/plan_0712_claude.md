## 0. 現状認識（正直な自己評価）

指摘は妥当です。これまで**機能の積み上げを優先し、細部の作り込み（配置・状態表示・
エラー文言・レイアウトの一貫性・レスポンシブ・固定ヘッダー/フッター）を後回し**に
してきました。結果として、画面ごとに品質のばらつきがあり、非エンジニアに伝わらない
表示（生の例外文字列など）が残っています。

根本原因（プロセス上の欠陥）:
1. **共通UIの土台（デザイントークン・共通コンポーネント）が未整備**で、画面ごとに
   その場しのぎのマークアップを書いていた。
2. **「完成の定義（Definition of Done）」にUI品質の観点が入っていなかった**
   （空状態・エラー文言・レスポンシブ・状態表示の確認項目が無い）。
3. **エラーは技術的にハンドリングしていたが、表示は非エンジニア視点で見直していない**。

本計画は、この3点を仕組みで是正します。

| Q5 | **フッターが無い** | アプリ共通フッター未実装 | `partials/footer.html`を新設（プロダクト名・バージョン・簡易リンク）。app-shellの最下部に固定表示 | `templates/partials/footer.html`(新規), 各画面, CSS | 中 |

### Q1 詳細（モデル連動）
- `MODEL_CHOICES`（プロバイダ→モデル配列）をテンプレートへJSON渡し、
  プロバイダ`<select>`の`change`で該当モデルだけを表示する軽量JS。
- **Ollamaはローカル導入モデルが任意**のため、ドロップダウンではなく
  「候補付きテキスト入力（datalist）」に切替。クラウドは候補ドロップダウン。
- サーバ側の検証（クラウドは候補内のみ許可・Ollamaは自由）は現行を維持。

### Q2/Q3 詳細（疎通確認の分かりやすさ）
- 例外分類ヘルパ `classify_llm_error(exc) -> (見出し, 原因, 対処)` を新設し、代表例を日本語化:
  - Ollama接続不可/404 → 「ローカルLLM(Ollama)に接続できません」「Ollamaが起動していない、または指定モデルが未導入の可能性」「`ollama serve`とモデル導入を確認」
  - 401/403（クラウド） → 「APIキーが無効か権限がありません」「キーの再確認」
  - キー未設定 → 「APIキーが未設定です」「環境変数の設定を確認」
  - タイムアウト → 「応答がタイムアウトしました」「時間をおいて再実行」
- 結果表示は**成功=緑バッジ＋応答抜粋、失敗=赤バッジ＋原因＋対処**を、設定カード内の
  専用パネルに。`404`のような生文字列は表示しない（詳細は折りたたみに退避）。

## 2. UI/UX品質ベースライン（全画面共通の底上げ）

個別修正だけでは再発します。**共通の土台**を整えます。

### 2.1 デザイントークンの確立
- 色・余白・角丸・影・タイポを`base.css`の`:root`に集約（既に一部あり）。
  - スペーシング階層（4/8/12/16/24/32）、フォントスケール、行間、影レベルを定義。
  - セマンティックカラー（success/warning/danger/info）を用途固定。
- 以後、画面固有CSSでの“その場の数値”を禁止し、トークン参照に統一。

### 2.2 共通コンポーネントの標準化
最低限、以下を「1つの正しい形」に統一（テンプレート`partials/ui/`＋CSS）:
- ボタン（primary/secondary/danger、サイズ、無効状態）
- フォーム（ラベル・入力・ヘルプ・エラー・必須表示）
- テーブル（ヘッダ固定・ゼブラ・空行）
- バッジ／ステータスピル（状態色の固定）
- カード／パネル
- アラート／トースト（success/warning/danger/info）
- **空状態（Empty State）**（「まだ○○がありません＋次アクション」）
- **ローディング状態**（ボタンのスピナー、領域のスケルトン）

### 2.3 メッセージ・エラー文言ガイド（非エンジニア向け）
原則:
1. **生の例外・HTTPステータス・スタックを画面に出さない**（詳細は折りたたみ or ログへ）。
2. **「何が起きたか＋なぜ＋次に何をすべきか」**の3点を日本語で。
3. 成功メッセージは**具体的**に（「保存しました」ではなく「LLM設定を更新しました」）。
4. 破壊的操作は**確認と結果**をセットで。

### 2.4 レイアウト規約
- **ヘッダー固定**（sticky）、**フッター常設**。
- コンテンツ最大幅・余白・セクション間隔の基準を統一。
- スクロール領域は本文のみ（ヘッダー/サイドバーは固定）。

### 2.5 レスポンシブ／アクセシビリティ（最低限）
- 主要ブレークポイント（〜640 / 〜900 / 900〜）で崩れないこと。
- キーボードフォーカスの可視化、コントラスト比、フォームラベルの関連付け。

## 3. 全画面レビューの方法論

「甘さが全体的」というご指摘に応えるため、**全画面を同一基準で棚卸し**します。

### 3.1 画面棚卸し
- 全URL/テンプレートを一覧化（ダッシュボード、チケット、分析、Copilot、レポート、
  ナレッジ、TPI、リスク、テスト計画、カレンダー、メンバー、自律運転、設定各種、管理各種）。

### 3.2 画面別チェックリスト（各画面を採点）
- [ ] ヘッダー固定・フッター表示
- [ ] ページタイトル/パンくずの一貫性
- [ ] 空状態の表示（データ0件時）
- [ ] エラー/メッセージが非エンジニア向け
- [ ] ボタン/フォーム/テーブルが共通コンポーネント準拠
- [ ] 状態表現（成功/警告/失敗/処理中）が適切
- [ ] レスポンシブ崩れなし（640/900）
- [ ] 破壊的操作に確認あり
- [ ] 導線（次にどこへ行くか）が明確
- [ ] ローディング/遅延時の表現

### 3.3 進め方
- 採点結果を表にし、**赤（要改修）から順に**共通コンポーネントで置換。
- 画面ごとにビフォーアフターをPlaywrightで確認。

## 4. PM基盤の方向性（先の市場調査より・案A/B/C）

UI品質と並行して、**プロダクトの核**をどう置くかの判断が要ります。
（調査レポート: 40+ツール・出典34件。WBS/ガントは汎用PMの標準装備）

- **案A（推奨・第一歩）**: 最小のPM基盤。自前の**WBS階層タスク＋ガント＋リスト/カンバン**。
  まず「見て計画できる」状態に。既存の同期チケットは参照として紐付け。
- **案B**: 案A＋**依存関係・クリティカルパス・ベースライン・ワークロード**
  （MS Project/OpenProject相当に近づける）。
- **案C（差別化）**: 汎用の後追いをせず、WBS/ガントは最小限に留め、
  **テスト計画・品質ゲート・ODC・自律運転と密結合**した検証工程特化の計画ビュー。

推奨は**案A→案C**（土台→検証特化）。**製品方向の判断のため、ここはご決定ください。**
