# PMO Agent — 設計ドキュメント

最終更新: 2026-07-11 / ステータス: **グリルセッション完了・Phase 1承認済み**

用語は [CONTEXT.md](../CONTEXT.md)（用語集）に従う。主要決定の経緯は [docs/adr/](adr/) を参照。

## 1. プロダクトビジョン

第三者検証会社のPMO業務を支援するAIエージェントシステム。**単一検証会社の社内ツール**として設計する（ADR-0001）。

PMOを「チケットの追い回し」から解放し、品質分析・診断・報告・顧客への提言といった高付加価値業務に集中させる。生成AIは分析の実行者・報告書のドラフター・相談相手（Copilot）として機能する。

## 2. ターゲットユーザーと課題

| ユーザー | 現状の課題 | 提供価値 |
|---|---|---|
| PMO担当 | JIRA/Redmineのチケット督促・状況確認に工数を吸われる | 停滞検知・督促通知・集計の自動化 |
| 品質分析担当 | ODC分析やメトリクス集計が手作業（Excel） | 自動分類（人が確定）・自動集計・所見生成 |
| 報告書作成者 | 週次/月次報告書の作成が属人的 | 書式準拠ドラフトの自動生成 |
| PMO全般 | 開発標準・過去知見の参照が個人の記憶頼み | RAGによる根拠付き回答・診断 |

## 3. 確定した設計方針

### 3.1 チケット連携ハブ
- **JIRAとRedmineの両方をPhase 1で実装**する。取込抽象化レイヤー（共通の正規化チケットモデル＋ソース別アダプタ）を最初から設計する
- **読み取り専用**。顧客のチケットシステムへの書き込み（督促コメント投稿等）は行わない。将来追加する場合も承認フロー必須
- 同期は数分間隔のポーリング（小規模前提・単一サーバー）。手動トリガーも提供
- 停滞条件は案件ごとに設定（例: N日更新なし・期限超過）。該当チケットを検知しシステム内通知を発行

### 3.2 品質分析エンジン
- **ODC分類はハイブリッド**: チケットのカスタムフィールドはマッピング設定で取込み、足りない軸はLLMがチケット本文から推定。**LLM推定は必ず人のレビューを経て確定**し、確定分のみ分析に使用（分類の出所と確定状態をデータとして保持）
- 定量メトリクス: 欠陥密度、検出フェーズ分布、収束曲線、再オープン率、滞留時間 等
- 診断: メトリクス＋ナレッジ（RAG）を入力にLLMが所見・リスク・提言を生成

### 3.3 レポート生成
- **システム内表示＋PDF出力**を正式サポート。書式はシステム側テンプレートで定義
- 既存Officeテンプレート（Excel/PPT）への書込みはスコープ外（将来オプションの設計余地のみ残す）
- 人間による編集・承認フローを前提（AIはドラフターに徹する）

### 3.4 RAG（ナレッジ基盤）
- 文書形式は混在（Office・Wiki・その他）→ **ファイルアップロードを基本とし、形式別パーサーを漸進追加**（優先順: PDF → Word → Markdown → Excel）
- 全社共通ナレッジと案件固有文書の2層スコープ
- 回答・診断には出典を明示

### 3.5 PMO Copilot チャット
- 案件のチケットデータ・分析結果・ナレッジを文脈に持つ相談チャット（案件単位のスレッド管理）
- 出典明示を必須とする

### 3.6 LLM抽象化層
- **LLMプロバイダ（OpenAI / Claude / Ollama）は案件単位の設定**（ADR-0002）。顧客との契約・NDAに応じて選択し、機密案件はOllamaでデータを社外に出さない
- APIキーは環境変数管理。プロンプト・応答の監査ログを保持

## 4. インフラ構成（ADR-0003）

- **PostgreSQL一本化**: DB＋pgvector（RAG）＋Procrastinate（非同期ジョブ）。追加ミドルウェアなし
- 規模前提: 同時〜10案件、各数百〜数千チケット、利用者〜20名、単一サーバー
- フロント: Djangoテンプレート＋最小JS。Copilotチャットのみストリーミング表示

```
┌─────────────────────────────────────────────────┐
│ Django (画面・API)                                │
│  accounts / engagements(旧projects) / dashboard  │
│  tickets / analytics / reports                   │
│  copilot / knowledge / llm                       │
└───────┬─────────────┬───────────────┬───────────┘
        │             │               │
  Procrastinate    LLM抽象化層      pgvector
  (同期・分析ジョブ) (案件設定で切替)  (ナレッジ検索)
        │
   JIRA / Redmine REST API（読み取り専用）
```

## 5. データモデル拡張案

- `tickets`: TicketSource(接続設定・案件単位) / Ticket(正規化チケット) / SyncRun(同期履歴) / StagnationRule(停滞条件) / Notification(督促通知)
- `analytics`: OdcClassification(チケット×ODC軸、出所と確定状態を保持) / MetricSnapshot(日次集計) / Diagnosis(診断)
- `reports`: ReportTemplate / Report(生成物・承認状態)
- `knowledge`: Document / DocumentChunk(pgvector埋め込み) / スコープ(共通・案件固有)
- `copilot`: ChatThread(案件単位) / ChatMessage(引用出典含む)
- `llm`: 案件のLlmProvider設定 / LlmCallLog(監査)

## 7. ロードマップ

- **Phase 0（済）**: 認証・案件選択・ダッシュボード骨格
- **Phase 1（承認済み・着手可）**:
  1. プロフィール編集・パスワード変更
  2. PostgreSQL＋Procrastinate移行
  3. JIRA＋Redmine両対応の読み取り専用同期（抽象化レイヤー含む）
  4. 停滞検知＋システム内通知
  5. UI用語を「案件」に統一
  6. 案件単位LLMプロバイダ設定の器
- **Phase 2 — 分析**: メトリクス集計→ダッシュボード実データ化、ODCハイブリッド分類（マッピング＋LLM推定＋人の確定）
- **Phase 3 — AI**: LLM抽象化層（3プロバイダ・案件単位切替）、Copilotチャット、レポート生成（画面＋PDF）
- **Phase 4 — RAG**: ナレッジ管理、pgvector検索、根拠付き診断・チャット強化

各Phaseはテスト（Claude担当）→実装（Codex担当）→レビュー→マージの分業フローで進める。

## 8. 残論点（実装中に解消する）

1. ODCマッピング設計のため、現場のJIRA/Redmineチケット項目（カスタムフィールド）のサンプルが必要
2. Ollama実行環境（GPUマシン）の確保はPhase 3までに
3. 報告書テンプレートの具体的な章立て・書式はPhase 3着手時にサンプル報告書をもとに設計
